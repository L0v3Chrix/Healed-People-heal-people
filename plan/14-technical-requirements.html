These rules exist so a neighbor's trust can never be lost to a bug.

This document serves every person who will ever scan a sticker — their safety is written into requirements, not left to good intentions.

Where the system begins and ends

System boundary

One responsive web platform — and the physical sticker is part of the system, not an accessory.

Worth Keeping is one responsive web platform with public recipient routes and authenticated operational workspaces. The physical sticker and its controlled code are part of the system.

Never trust the browser

The platform must not trust the browser to authorize inventory, redemption, financial, claims, role, or audit transitions.

Nine roles, each with hard limits

Required roles

Every role gets exactly what it needs — and an explicit list of what it may never do.

Access rules that never bend

Operator, finance, and organization-owner accounts require MFA. Recipient first scan requires no account. Distributor access must support partner assistance and a documented non-digital fallback.

Three state machines that keep the story straight

Required state machines

Every sticker, offer, and redemption moves through named states — nothing slips between them.

Sticker inventory

draft code → minted → print_exported → print_verified → activated → packed → assigned → passed → claimed → redeemed

  • Exception states: quarantined, void, lost, stolen, damaged, recalled, expired.
  • A terminal or exception transition must record actor, timestamp, reason, prior state, and correlation ID.
  • Printed codes remain unusable until batch activation.

Offer and claim

Offers: draft → pending_approval → active → paused → retired

Successful claims: revealed → claimed → redemption_pending → redeemed, with expired, cancelled, disputed, and reversed as controlled outcomes.

Claim creation must atomically enforce availability, active dates, recipient/device abuse controls, and per-offer caps.

Redemption and participant value

initiated → verified → obligation_created → under_review → approved → execution_pending → executed → reconciled

  • Exceptions: verification_rejected, disputed, reversed, execution_failed, reissued, cancelled_by_authority.
  • A verified redemption may produce exactly one obligation set under the versioned approved formula.
  • Corrections use compensating/reversal entries; financial history is never overwritten.

What each part of the platform must do

Functional requirements

Five surfaces, one standard: clear for the person in front of it, accountable behind it.

Public recipient experience

  • Resolve a high-entropy public sticker code without exposing internal sequential IDs.
  • Explain the giver-first model before requesting optional profile data.
  • Display offer value, business, location, expiry, restrictions, accessibility, and redemption instructions before claim confirmation.
  • Support a low-bandwidth route with compressed media and no required animation.
  • Use granular consent for personalization and marketing; allow skip, withdrawal, export, and deletion where applicable.
  • Never expose distributor identity or expected participant value to the recipient.

Business and verification

  • Provide approved offer templates and a custom-offer submission path.
  • Require authenticated location staff or a controlled short-lived location session to verify redemption.
  • Do not store or expose a shared plaintext/PIN-equivalent secret in sponsor-readable offer data.
  • Accept an idempotency key for every verification attempt.
  • Record location, staff account, claim, timestamp, result, and correction lineage.
  • Provide an offline exception procedure; offline events remain pending until authenticated synchronization and review.

Partner/distributor operations

  • Record approval, training completion, consent version, accommodations, status, assigned partner, and support preference.
  • Transfer packs through counted custody events with sender/receiver acknowledgement.
  • Show the distributor plain-language counts for assigned/passed/claimed/redeemed activity and approved/executed value.
  • Provide a human support channel and partner-assisted statement when no reliable device exists.
  • Keep safety/incident details segregated from marketing and business reporting.

Finance and accountability

  • Version every formula, cap, fee/overhead rule, sponsor restriction, and effective date.
  • Create obligations within the same database transaction as the verified redemption or through a transactionally safe outbox worker with idempotent replay.
  • Require two distinct authorized actors for material approval and execution evidence.
  • Reconcile obligations, approved batches, provider/manual execution evidence, failures, reversals, and bank/accounting totals.
  • Generate participant statements and aggregate sponsor/impact reports without disclosing protected information.

Operator and support

  • Provide queues for pending offers, print batches, assignments, redemption exceptions, fraud flags, disputes, participant-value reviews, failed executions, incidents, consent/deletion requests, and claims approval.
  • Every administrative action requires a reason and appears in an append-only audit log.
  • Provide safe impersonation only if later approved; beta default is no impersonation.

The data the system must keep — and how

Data requirements

Keep what the fourteen-table draft got right, then add the tables accountability demands.

Retain the useful concepts in the existing fourteen-table draft while revising names and relationships after professional determinations. Required additions include:

  • Organization, membership, role, location staff, and account lifecycle tables.
  • Partner organization and partner–distributor relationship.
  • Consent/version records and data-subject request records.
  • Inventory custody events and batch print/QA evidence.
  • Idempotency keys, state-transition log, outbox/job records, and immutable audit events.
  • Formula/version, obligation, obligation line, approval, execution, reconciliation, and correction records using the approved public terminology.
  • Support cases, incidents, appeals, and restricted notes with separate access policies.
  • Claims/evidence register for public reports.
Formats that prevent quiet errors

Financial amounts use integer cents plus ISO currency, never floating point. Timestamps use UTC and render in the user’s locale. Each transactional row uses UUIDs, created/updated metadata where mutable, and immutable event IDs where append-only.

Ten invariants the database must never break

Database invariants

These are enforced in the database itself — not politely requested of the application.

  1. One code, one active sticker

    A public sticker code maps to no more than one active sticker unit.

  2. One sticker, one active claim

    One sticker unit can have no more than one active claim.

  3. One claim, one verified redemption

    One claim can have no more than one effective verified redemption.

  4. One redemption, one obligation set

    One effective verified redemption can have exactly one obligation set for one formula version.

  5. Replays return the original

    Replays with the same idempotency key return the original result.

  6. Caps never go negative

    Offer caps are consumed atomically and never become negative.

  7. Lines balance to the formula

    Obligation lines balance to the approved formula total.

  8. Executed never exceeds approved

    Executed value cannot exceed approved value without a new approval.

  9. Corrections preserve the original

    Reversal/correction entries reference their source and preserve the original.

  10. Cross-org access denied by default

    Cross-organization access is denied by default and tested in PostgreSQL.

Where the API draws its lines

API boundary

Every request is validated on the server; every failure tells the operator enough and an attacker nothing.

  • Server-side route handlers or server actions validate authentication, authorization, schema, state, and idempotency.
  • Service-role/database-owner credentials never reach the browser.
  • Public endpoints are rate-limited, abuse-monitored, and return non-enumerable errors.
  • Webhooks, if later added, require signature verification, timestamp tolerance, deduplication, replay tooling, and dead-letter handling.
  • Error responses expose a correlation ID, not stack traces or protected details.

How we protect people and their data

Security and privacy

Security here is a care practice — the people this serves have the most to lose from carelessness.

  • Organization-owned password manager and MFA for privileged accounts.
  • Separate local, preview/staging, and production environments and secrets.
  • Least-privilege RLS with positive and denial tests for every role/table combination.
  • Dependency pinning, automated vulnerability/dependency updates, secret scanning, and protected main branch.
  • Content Security Policy, secure cookies, CSRF protection where applicable, input/output encoding, upload restrictions, and rate limiting.
  • Written incident classification, notification, containment, evidence preservation, recovery, and postmortem process.
Anonymous first, always

Anonymous-first scanning; do not collect name, exact age, hobbies, precise location, or device fingerprint unless a documented purpose, consent/legal basis, retention, and access rule exists. No sale of personal data. Aggregate campaign/business measurement must not allow participant re-identification.

Everyone gets in the door

Accessibility and content

If a workflow is beta-critical, it works for keyboards, screen readers, low-end phones, and people helping in person.

  • Target WCAG 2.2 AA for all beta-critical workflows.
  • Keyboard navigation, visible focus, semantic landmarks, labels, error summaries, 200% zoom/reflow, sufficient contrast, reduced motion, captions/transcripts, and non-color status cues.
  • Plain-language English at beta; Austin-language needs are assessed with the operating partner before launch. Critical participant materials require partner-assisted explanation and printable versions.
  • Support current and previous major versions of Safari iOS, Chrome Android, Chrome desktop, Safari desktop, Edge, and Firefox at beta launch.

How reliable it has to be

Reliability and operations

Concrete numbers, so "it works" is a measurement instead of a mood.

99.5%Monthly availability, beta-critical web/API routes
<1sTransaction API p95 under expected pilot load
<3sPublic scan page usable content, low-end mobile/4G
4 hrsRecovery time objective, beta-critical service
  • Production availability objective: 99.5% monthly for beta-critical web/API routes, excluding announced maintenance.
  • Transaction API target: p95 below 1 second under expected pilot load; public scan page usable content target below 3 seconds on a representative low-end mobile/4G profile.
  • Recovery point objective: 24 hours maximum for general content and 15 minutes for transactional/financial records where the selected platform supports it.
  • Recovery time objective: 4 hours for beta-critical service.
  • Daily automated backups plus pre-migration backup; quarterly restore test during beta preparation and before major schema changes.
  • Structured logs, correlation IDs, error monitoring, availability checks, database health, job/queue failure alerts, and cost alerts.

Proof before people

Test requirements

Seven layers of testing stand between the code and a real participant.

🧪Unit tests

Formulas, permissions, state transitions, token/code utilities, and consent logic.

🗄️PostgreSQL tests

RLS, invariants, atomic cap use, idempotency, ledger/obligation creation, reversals, and forbidden mutation.

🔌API integration tests

Every success, replay, authorization denial, invalid transition, dispute, and failure path.

🔁End-to-end tests

Recipient scan/claim, staff redemption, partner assignment, finance review, correction, report, and no-smartphone support workflow.

📱Physical QR test matrix

Across print proofs, glare, distance, damage, older phones, and low light.

♿Accessibility checks

Automated checks plus keyboard and screen-reader manual review.

🛡️Security review

Security review and tabletop incident/recovery exercise before real participant launch.

The last gate before real people

Technical release gate

Production beta deployment happens only when every one of these is true.

  • All critical tests green.
  • Zero unresolved critical/high security findings.
  • Verified backups and restore.
  • Privacy/accessibility review.
  • Operator runbooks.
  • On-call ownership.
  • Migration rollback.
  • Signed operational readiness from product, engineering, partner/safety, finance, and privacy/security owners.

Nothing on this list is ceremony. Each line is a promise kept before a single real sticker changes hands.
