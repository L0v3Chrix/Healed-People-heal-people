One app, one database, one honest path to launch

Operating recommendation

The whole beta lives in one organization-owned application, so nothing important belongs to only one person or one prototype.

Consolidate the beta into one organization-owned Next.js application backed by Supabase/PostgreSQL and deployed through GitHub to Vercel. Preserve the story site as campaign content or migrate its approved sections into the unified application.

🧑‍⚖️Money stays human-reviewed

Keep financial execution human-reviewed in beta. Software records the decision; people make it.

⏸️AI and POS wait their turn

Defer production AI and POS integrations. The core proof does not need them yet.

No layer is a trap

Every recommendation above comes with an export or fallback route. If a vendor changes, the community's data and the program's history come with us.

Pin versions so releases stay boring

Version policy

Nothing in this stack is allowed to change under our feet — upgrades happen on purpose, with tests, or not at all.

  • Replace every latest dependency with an exact version that matches the verified lockfile baseline.
  • Use Node.js 22 LTS for development and CI until a tested migration is approved.
  • Commit one lockfile per application; do not regenerate it casually.
  • Apply security patches through pull requests with automated tests; schedule framework/minor upgrades rather than allowing implicit drift.
  • Record production database extensions and CLI versions in repository documentation.

Four environments, one rule about real data

Environment model

Real people's information only ever lives where it is genuinely needed — production — and nowhere else.

Hard boundary on real data and secrets

No production database copy may be used in local/preview. Secrets are stored in platform environment stores and an organization password manager, never in source or chat.

Six decisions that keep the system trustworthy

Architecture decisions

These are the load-bearing choices — each one exists to protect a promise the program makes to people.

🗄️1 · One source of truth

PostgreSQL owns state; UI projections never create financial truth.

🎭2 · Anonymous-first recipients

Optional profile data is separable from claim/redemption.

🛡️3 · Server-authorized writes

Clients request transitions; server/database validates them.

📜4 · Append-only accountability

Inventory custody, state transitions, audit, obligations, executions, and corrections preserve history.

🔌5 · Provider adapters

Email, campaign data, accounting export, and future payment providers sit behind small interfaces.

🧘6 · No premature complexity

Pilot scale does not justify microservices, Kafka, blockchain, or multiple databases.

Simple enough to trust, honest enough to audit.

What we said no to, and why

Alternatives considered

Each of these was weighed seriously — and set aside for a reason we can say out loud.

  • Firebase — rejected as primary because the existing relational/ledger/RLS design and SQL tests favor PostgreSQL.
  • Custom Express API — deferred; Next.js server routes are sufficient at pilot scale and reduce deployment surface.
  • Native mobile apps — deferred; responsive web and partner-assisted access cover the approved beta.
  • Stripe Connect or payout provider now — deferred until participant-value classification, identity, and custody are approved.
  • Third-party behavioral ad platform — excluded because the concept promises not to sell personal data and should retain control of sensitive measurement.
  • Full AI personalization — deferred due privacy, moderation, cost, reliability, and lack of necessity for the core proof.

Spend carefully, and never get surprised by a bill

Cost-control rules

Every dollar this project spends on tooling should be planned, owned, and watched.

  1. Start free where it's honest

    Begin on free/developer tiers only where terms permit real beta use; budget production tiers before launch.

  2. Give every paid service an owner

    Assign account owner, billing owner, backup admin, monthly ceiling, alert threshold, export route, and cancellation procedure for every paid service.

  3. Never launch on one person's account

    Do not launch a campaign or beta on personal-only accounts with no backup administrator.

  4. Review before limits bite

    Reassess service tiers at 50%, 75%, and 90% of included usage.
