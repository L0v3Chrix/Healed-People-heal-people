This document serves the original creators — their recovered work was read, re-read, and verified before a single conclusion was drawn.

Recover the real files — the recovery boundary

Where the reviewed source actually came from

The local Dropbox File Provider exposed many zero-block dataless placeholders, so review could not trust the workspace copy alone.

The linked personal Dropbox account was already authenticated in Chrome, so the following folders were downloaded separately and extracted into an isolated temporary directory rather than overwriting the workspace:

  • src/
  • supabase/
  • docs/
  • public/
  • deliverables/
Both sources verified before review

All five ZIP archives passed integrity testing before review. The nested story project was reviewed from an authoritative private Git recovery at the workspace-recorded revision.

Three review passes, one honest picture

How the corpus was read

The recovered material was reviewed in three deliberate passes, each with a different question in mind.

  1. Pass 1 — Intent and inventory

    Read the complete recovered text/source corpus and inspected the visuals to identify audiences, product promise, role flows, entities, prototype behavior, and the difference between concept, mock, and implemented capability.

  2. Pass 2 — Consistency and requirements

    Cross-searched and reread the corpus for operational and financial commitments, including micro-grant, payout, staff PIN, overhead, review windows, retention, pilot quantities, reward weights, nonprofit/fiscal-sponsor language, partner claims, manual versus instant handling, service-role access, and illustrative labels.

  3. Pass 3 — Production scrutiny

    Reviewed the implementation for security, money integrity, privacy, fraud controls, accessibility, responsive behavior, evidence quality, and test coverage.

Key outcome of Pass 1

The public prototype, static explainer, technical roadmap, schema, and story site are different maturity layers. None alone is an approved production specification.

Six conflicts the interview must settle

Pass 2 findings — where the story disagrees with itself

These conflicts require interview decisions before any of the affected copy, schema, or numbers can be treated as requirements.

  1. The pledge graphic says the distributor receives a micro-grant instantly; the roadmap and explainer describe manual payout after review.
  2. The roadmap claims redemption creates two pledge-ledger rows, but the migration contains no redemption-to-ledger trigger.
  3. “Micro-grant” is embedded in public copy and schema even though classification, tax treatment, payout rails, and legal structure are unresolved.
  4. Staff PIN, 20% overhead, seven-day review, 90-day retention, pilot quantities, and 70/20/8/2 rarity weights are proposals or fixtures—not approved requirements.
  5. Four separately named community/hero PNGs are exactly the same file.
  6. The marketing prototype presents dashboard and Studio behavior that is client-only and not persisted.

What production readiness is still missing

Pass 3 findings — notable readiness gaps

The implementation review surfaced gaps that must close before the system can be trusted with real money and real people.

  • The service-role route layer is the central authorization boundary but is not present in the recovered app source.
  • The database does not enforce several roadmap claims: atomic offer-cap decrement, two-way ledger split creation, per-redemption split uniqueness, distributor identity/authentication, payout-to-ledger foreign keys, event idempotency, audit logging, retention, or daily redemption ceilings.
  • Current tests do not exercise sponsor self-read RLS, race conditions, service-role authorization, ledger correction/reversal rules, fraud freezes, or retention.
  • Sponsor-readable offer rows include confirm_pin_hash, which needs threat-model review.
  • Static sites load Google Fonts remotely; the main app has reduced-motion and focus treatments, while the roadmap has weaker explicit keyboard-focus treatment.
  • Generated visuals contain exact-looking metrics and “instant” financial claims that cannot be used as production evidence.

Run it in a sandbox — isolated executable verification

Builds and syntax checks, nothing overwritten

Verification used temporary copies and did not overwrite the Dropbox workspace.

  • Main application: production build passed after the temporary harness supplied the @/* source alias unavailable from the unread original tsconfig.json; TypeScript passed and / prerendered.
  • Story application: clean lockfile install and production build passed; / prerendered and /api/oracle built as a dynamic route.
  • Static explainer JavaScript: syntax pass.
  • Static technical-roadmap JavaScript: syntax pass.
  • Database verification shell script: syntax pass.
Two caveats from the runs

Story build warning: missing metadataBase causes social metadata URLs to fall back to localhost during build. Supabase migrations/tests: not executed because the current machine has no callable Docker, Supabase CLI, or local Postgres test environment.

What these checks do NOT prove

These checks prove build/syntax viability only. They do not prove the marketing mocks are implemented product workflows, the ledger invariants are complete, the database is deployed, or the operating/legal decisions are approved.

Name what stayed unreadable — remaining placeholders

17 files still blocked, and why

A small set of workspace files never materialized locally, and the log records exactly what was tried and what was not.

The root README, Brand Board, four root build/config artifacts, lockfile, and ten historical screenshots remain zero-block dataless placeholders. They are recorded as 0/3 in the manifest and are excluded from final-plan claims until readable.

Native Dropbox/Finder recovery audit

  • Finder/Dropbox file information confirmed the 2.65 KB root README is Online-Only, uses 0 bytes of disk space, and is expected to download when opened.
  • Finder continued to label the target files Downloading, but opening them did not materialize any blocks and the item action menu remained disabled during the false download state.
  • The Dropbox desktop process was given a normal quit/relaunch. macOS also performed a normal quit/restart of the user File Provider daemon after it had remained at unusually high CPU and multi-gigabyte memory usage.
  • The restarted provider immediately returned to high CPU and the same target files remained zero-block dataless placeholders.
  • A read-only fileproviderctl check against the Dropbox-Personal domain produced no output after more than one minute and was stopped. No repair, forced provider-extension termination, credential access, deletion, overwrite, or broad Dropbox mutation was attempted.

Current conclusion: the remaining 17 files are blocked by the local Dropbox/File Provider state, not by missing filenames or an unissued download request. The isolated web recovery remains the authoritative source for the reviewed core application.
