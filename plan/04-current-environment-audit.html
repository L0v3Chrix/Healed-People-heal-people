Proving the builds work — without touching production

Isolated verification in temporary directories

The recovered source was copied to temporary directories so verification would not edit or hydrate the production workspace implicitly.

Main marketing application

  1. Fresh resolution

    A fresh dependency resolution from the root package.json installed 362 packages and selected Next.js 16.2.10 because the manifest uses latest.

  2. Expected first failure

    The first build correctly failed because the original tsconfig.json remained an unreadable Dropbox placeholder and the @/* path alias was therefore absent from the isolated copy.

  3. Successful build

    After adding a temporary standard Next.js TypeScript configuration containing only the source-required @/* → ./src/* alias, the recovered application compiled, passed the Next.js TypeScript phase, and prerendered / successfully.

What this proves — and what it doesn't

This proves the recovered application source is buildable. It does not prove the unread root configuration matches the temporary harness, nor that latest is a safe production version policy.

Story application

  • A clean npm ci from the authoritative story lockfile installed 28 packages
  • npm run build compiled, passed TypeScript, prerendered the story page, and built the dynamic /api/oracle route
One readiness warning

The build emitted one readiness warning: metadataBase is missing, so social-image URL resolution falls back to http://localhost:3000 during build.

Static and database tooling

  • Both static deliverable JavaScript files pass node --check
  • scripts/db-verify.sh passes bash -n syntax validation

Database migrations were not executed in this audit because neither Docker nor a callable local Postgres/Supabase environment is currently available. Static SQL review is not equivalent to a migration/test run.

Decisions only the interview can make

Eight open questions the setup guide must wait for

The final installation/setup guide cannot responsibly choose these until the operating model is confirmed.

  • Whether beta needs local Docker/Supabase development or can use a managed-only workflow
  • Whether Vercel CLI is required or dashboard/Git deployments are sufficient
  • Whether Gemini/Sticker Oracle remains in the beta
  • Which payments or disbursement provider, if any, is legally approved
  • Which email/SMS, analytics, monitoring, support, accounting, consent/research, campaign, and fulfillment platforms are required
  • Whether design/video/print-production tools are owned by the core team or outside vendors
  • Which accounts are organization-owned rather than personally owned
  • Required access, billing owner, backup administrator, offboarding, export, and disaster-recovery rules

The bar every approved tool must clear

Nine checks before a tool enters the final guide

For every tool that the interview approves, the final guide will include all nine of these.

  1. Justified need

    Why the tool is required by a confirmed workflow

  2. Supported install path

    Supported installation path for this Apple Silicon Mac or the relevant hosted platform

  3. Version policy

    Exact version or version policy

  4. Named owners

    Account owner and billing owner

  5. Secret-safe setup

    Setup/configuration steps without exposing secrets

  6. Proof it works

    Verification command or acceptance check

  7. Access and backup

    Access-control and backup/export expectations

  8. Learning path

    Training owner and learning resource, when a skill gap exists

  9. Clean exit

    Removal or rollback instructions when the tool changes system state

The spirit of this audit: know exactly what's on the workbench before promising what we'll build with it. An honest inventory today saves the community from a broken promise tomorrow.
