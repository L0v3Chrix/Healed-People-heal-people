How to read each status

The honesty scale for every decision

Every row in this register carries one of five statuses — and a decision only moves up the scale when a named owner says so.

Confirmed
Explicitly authorized in the interview with an owner
Provisional
Usable for planning but still requires named confirmation
Professional determination required
Legal, accounting, benefits, safety, privacy, or other qualified review controls the answer
Open
Source material contains a proposal, fixture, mock, or conflicting positions
Not applicable
Explicitly excluded by the authorized owner
The default is open

All entries currently remain open unless the interview record says otherwise. Nothing in this register is settled by silence.

Seven implementation gaps that are not product decisions

Evidence-backed, and not to be fixed silently

These are technical facts about the current codebase and assets — they should not be “resolved” by choosing a product rule silently.

  1. Missing ledger trigger

    The migration has no redemption-to-ledger trigger despite the roadmap describing automatic two-row ledger creation.

  2. Unenforced invariants

    Offer-cap consumption, ledger split uniqueness, event idempotency, and several financial/state invariants are not database-enforced.

  3. Exposed PIN hash

    confirm_pin_hash is exposed through sponsor-readable offer rows and requires threat-model review.

  4. Absent authorization boundary

    The service-role route layer described as the authorization boundary is absent from the recovered main application.

  5. Duplicate visual files

    Four differently named community/hero visual files are identical.

  6. Conflicting generated claims

    Generated graphics contain operational and financial claims that conflict with the roadmap.

  7. Unstable dependency policy

    The root project’s latest dependency policy prevents a stable, reviewable production baseline.

How a decision becomes confirmed

The update rule

There is exactly one path from open to confirmed, and it runs through a named human.

When an interview answer is accepted, update the corresponding row with the answer, owner, evidence, date, and status. A source proposal never becomes “Confirmed” merely because it already appears in code, seed data, a visual, or public copy.
